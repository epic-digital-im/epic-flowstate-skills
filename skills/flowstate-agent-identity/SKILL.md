# Agent Identity

**Status:** Active (updated 2026-05-02 for D1-as-source-of-truth)
**Purpose:** Every subagent MUST assume a defined agent identity — no anonymous work, no unattributed entities
**Scope:** All subagent dispatches, entity creation, document creation, time tracking, task claims
**Trigger:** Before dispatching any subagent or creating any entity on behalf of an agent
**Input:** Agent slug (matches `teammembers.slug` in D1; same value as the `name` field in `.flowstate/agents/<slug>.json`)
**Output:** Fully identified agent context with teamMemberId for entity attribution

---

## Source of Truth

```
D1 teammembers collection IS the runtime source of truth for agent identity.
Local files (.flowstate/agents/*.json, .claude/agents/*.md) are
definitions / generated artifacts:

  .flowstate/agents/<slug>.json   →  bootstrap definition; pushed to D1
                                     by `flowstate sync-agents`. Editable.
                                     Not read at runtime.

  .claude/agents/<slug>.md        →  generated from teammembers data by
                                     ClaudeAgentGenerator for Claude Code
                                     consumption. NEVER edit by hand.

  teammembers (D1 RxDB)           →  authoritative. Looked up by slug.
                                     teamMemberId = record.id.
```

After PR #272 + #275: identity resolution happens via `mcpToolCall('collection-query', { collection: 'teammembers', selector: { slug: name } })` and TS-side filtering on `isAgent` + `_deleted`. No CLI command reads `.md` or `.json` files at runtime.

---

## Iron Law

```
NO SUBAGENT OPERATES WITHOUT A LOADED AGENT IDENTITY.
NO ENTITY IS CREATED WITHOUT userId MATCHING THE AGENT'S teamMemberId.
NO RUNTIME COMMAND READS .claude/agents/<slug>.md OR .flowstate/agents/<slug>.json
TO RESOLVE A teamMemberId. QUERY teammembers.slug INSTEAD.
```

## Primary Session Identity

The primary Claude Code session (the orchestrator) is NOT a subagent and does not get dispatched via the Task tool. It still creates entities (discussions, approvals, documents) and must use a real identity.

**The primary session uses the coordinator agent — resolved via slug:**

```
Agent slug:     coordinator
Lookup:         flowstate agent status coordinator   (or MCP collection-query)
teamMemberId:   record.id from the resolved teammembers row
characterName:  record.userName  (used as `userName` on discussions)
```

At the start of every session that creates FlowState entities:

1. Resolve the coordinator agent: `flowstate agent status coordinator` (or programmatically via `resolveAgentRecord('coordinator')`)
2. Use `record.id` as `userId` on all entities
3. Use `record.userName` (e.g. `"FlowState Coordinator"`) as `userName` on discussions

This applies to ALL entity creation from the primary session, including discussions, approvals, documents, and activities.

**If a human user is the actor** (e.g., approving something), use the human's `teamMemberId` from their `users`/`teammembers` record, not the coordinator's.

## Where agent data lives

Five artifacts, four roles:

| Artifact | Purpose | Written by | Read by |
|---|---|---|---|
| `.flowstate/agents/<slug>.json` | Bootstrap definition (description, instructions, tools, model, RACI). The `name` field becomes `slug` in D1. | Operators (humans) | `flowstate sync-agents` (push to D1) |
| `teammembers` (D1 RxDB) | **Authoritative agent identity**. Slug-keyed; holds `metadata.openclawConfig`, runtime status, device key fallback, `extended.agent` (snapshot of original .json). | CLI commands (`set-model`, `provision*`, `start`, `stop`, `chat`); `AgentSyncManager`; orchestrator | Every runtime resolution via `resolveAgentRecord(slug)` |
| `.claude/agents/<slug>.md` | Generated frontmatter for Claude Code subagent dispatch. Carries `<!-- @generated -->` header. | `ClaudeAgentGenerator` (regenerated by `sync-agents`) | Claude Code's Task tool |
| `volumes/agents/<slug>/openclaw-data/workspace/*.md` | **Runtime identity surface inside the openclaw container**. 6 files: IDENTITY / SOUL / USER / TOOLS / AGENTS / HEARTBEAT. Rendered from teammember metadata + `extended.agent`. | `flowstate agent provision` (writes 6); openclaw entrypoint (writes REPOS.md heredoc) | The agent itself, on every container boot |
| Vault item `flowstate-admin/agents/<slug>/openclaw-device-identity` | Ed25519 private key for OpenClaw connect-frame signing. | `flowstate agent provision-device-key` | `flowstate agent chat` (via `readOpenClawDeviceIdentity`) |

The 100+ agent fleet inventory lives in D1 — `flowstate agent list` is the canonical browser. `.flowstate/agents/*.json` may have more or fewer files than D1 has rows; `flowstate sync-agents --dry-run` tells you the exact diff.

### The 6 workspace markdown files (in-container identity surface)

When an openclaw sidecar boots, the agent reads these files from `/home/openclaw/.openclaw/workspace/` (mounted from `volumes/agents/<slug>/openclaw-data/workspace/` on the host). These ARE the agent's runtime self-knowledge:

| File | Source data | Purpose |
|---|---|---|
| `IDENTITY.md` | `userName` (characterName), `extended.agent.metadata.orgRole`, `characterEmail`, `tone`, `bio` | Name, role, email, vibe, emoji, short bio. Read on every wake to confirm "who am I". |
| `SOUL.md` | `extended.agent.instructions`, `metadata.backstory`, `metadata.directReports`, `metadata.raciAssignments` | Personality + system prompt + RACI. The agent's operating-principles document. |
| `USER.md` | `orgId`, top-level `workspaceId`, `id` (teamMemberId), `metadata.productId` | Org context + the agent's own IDs (used to scope every CLI/MCP call). |
| `TOOLS.md` | Same IDs as USER.md + role | flowstate CLI usage patterns + auth flow + queries the agent will make every turn. |
| `AGENTS.md` | `metadata.directReports`, `orgId`, top-level `workspaceId` | Team coordination handshake (task → discussion → completion ack pattern). |
| `HEARTBEAT.md` | `metadata.orgRole` keyword match (CEO/CTO/PM/engineer/QA) | Periodic self-check prose tailored to the role. |

REPOS.md is **not** in this list — the openclaw entrypoint owns it (heredoc rewrite on every boot). The other 6 are written once at provision time and only re-rendered on explicit re-provision. Provision-time content is preserved across container restarts.

**Implication for subagents:** when an openclaw agent dispatches a task, it should read its OWN `IDENTITY.md` to know its `userName` for entity attribution — but the runtime canonical source is still the teammember row. Cross-check by calling `flowstate agent status <slug>` (slug = the agent's own slug, available in `IDENTITY.md` Role field).

## Loading Agent Identity

### Step 1: Resolve by slug

**Via the CLI helper** (preferred for code; `resolveAgentRecord` is exported from `agent/utils/lifecycle.ts`):

```ts
import { resolveAgentRecord } from '@epicdm/flowstate-cli'
const agent = await resolveAgentRecord('senior-frontend-engineer')
// → { id, userName, orgId, metadata }
```

**Via the CLI binary** (for shell scripts / OpenClaw sidecars):

```bash
flowstate agent status senior-frontend-engineer --json
# → { id, userName, orgId, metadata.containerStatus, metadata.openclawDeviceIdentity, ... }
```

**Via direct MCP** (if you control the orgId/workspaceId injection):

```js
mcp__epic-flowstate__collection-query({
  collection: 'teammembers',
  selector: { slug: 'senior-frontend-engineer' },
  orgId: '<active org>'
})
// → { documents: [<row>] }   filter isAgent=true + _deleted=0 client-side
```

### Step 2: Extract Critical Fields

From the resolved record:

| Field | Source | Purpose |
|---|---|---|
| `id` | top-level | **CRITICAL** — `teamMemberId`. Use as `userId` on every entity. |
| `userName` | top-level | Display name (e.g., "Elena Vasquez") for discussions/activities |
| `orgId` | top-level | Org-scope guard (subagents must operate within their org) |
| `slug` | top-level (post-#272) | Stable handle. Same as the `--name` flag passed to CLI commands. |
| `metadata.openclawConfig.models` | metadata | Active model + route (default, fallback, routeModel) |
| `metadata.containerStatus` | metadata | running / stopped / unknown — runtime state |
| `metadata.openclawDeviceIdentity.deviceId` | metadata | SHA-256 fingerprint of the agent's Ed25519 public key |
| `extended.agent.instructions` | extended (when synced) | System prompt body — the agent's personality |
| `extended.agent.metadata.characterName` | extended.agent.metadata | Same as `userName`, kept for provenance |
| `extended.agent.metadata.orgRole` | extended.agent.metadata | Role title (e.g., "Senior Frontend Engineer") |
| `extended.agent.metadata.characterEmail` | extended.agent.metadata | Used as the git author email when committing on the agent's behalf |

`extended.agent` mirrors the original `.flowstate/agents/<slug>.json` content. It's snapshotted into D1 by `sync-agents`; refer to D1 for the live copy, not the file on disk.

### Step 3: Use teamMemberId on every entity

`record.id` IS the `teamMemberId`. Use it directly:

```
Resolved record (D1 teammembers WHERE slug='senior-frontend-engineer')
  └── id: "team_BySuGAjoaD"
        └── Used as userId on ALL entities this agent creates
```

**If `resolveAgentRecord` throws `Agent slug "<x>" not found`:** STOP. Do not dispatch. Either:
- The agent's `.flowstate/agents/<x>.json` was never synced (run `flowstate sync-agents`).
- The slug is mistyped — `flowstate agent list` shows valid slugs.
- The active server's workspaceId doesn't match where the agents live (project misconfig).

**If the resolved record has `isAgent=false`:** STOP. The slug matched a human teammember, not an agent.

## Tool Awareness — Agent Inbox

After identity is resolved (`teamMemberId` known via `slug` query), the agent's first action on each heartbeat should be:

```text
mcp__epic-flowstate__agent-inbox { "slug": "<my-slug>" }
```

This single call replaces the historical 2–3 chained `collection-query` calls used to find:

- Tasks assigned to me with `status ∈ {Planned, In Progress}`
- Discussions in the last 7 days where `@<my-slug>` was written
- Replies on threads I authored, OR replies that mention me

**Returned shape:** `AgentInboxResult` with three independent arrays (`tasks`, `mentions`, `replies`) plus `generatedAt` and `windowStart`. See `docs/guides/agent-inbox-user-guide.md` in the `epic-flowstate-community` repo for the full type reference.

**Hybrid mention match:** every mention stores BOTH the literal `identifier` and the resolved `teamMemberId`. The query layer matches on either, so a slug rotation does not break older mentions. Agents do not need to handle this — it's transparent.

**Forward-only:** the parser hook went live 2026-05-03; older discussions stay `mentions = NULL` and are silently skipped. The inbox empty-state explicitly discloses this.

**Replacement for prior practice:** if a prior version of this skill (or skill code) instructs agents to chain queries like:

```
collection-query tasks {assigneeId: <teamMemberId>, status: ...}
collection-query discussions {createdAt: >=, mentions: ...}
collection-query discussions {parentId: ..., ...}
```

prefer `agent-inbox` instead. It's cheaper end-to-end and handles the hybrid mention match + reply two-pass walk correctly.

**CLI equivalent (operator-side):** `flowstate agent inbox <slug>` — markdown by default, `--json` for raw, `--since 24h` to window down.

## Entity Attribution Rules

Every FlowState entity created by an agent MUST have `userId` set to the agent's `teamMemberId`:

| Entity Type   | userId Field           | userName Field                 | Example                         |
| ------------- | ---------------------- | ------------------------------ | ------------------------------- |
| Documents     | `userId` on create     | N/A                            | `userId: "team_BySuGAjoaD"`     |
| Tasks (claim) | `assigneeId` on update | N/A                            | `assigneeId: "team_BySuGAjoaD"` |
| Discussions   | `userId` on create     | `userName` (required, from characterName) | `userId: "team_BySuGAjoaD", userName: "Elena Vasquez"` |
| Activities    | `userId` on create     | `userName` (required, from characterName) | `userId: "team_BySuGAjoaD", userName: "Elena Vasquez"` |
| Time Entries  | `userId` on create     | N/A                            | `userId: "team_BySuGAjoaD"`     |
| Approvals     | `userId` on create     | N/A                            | `userId: "team_BySuGAjoaD"`     |

**Discussion userName rule:** The `userName` field on discussions is **required** by the schema. It must be the agent's `metadata.characterName` (e.g., "Elena Vasquez"), never "Claude", "claude-code", or the kebab-case agent name.

### MCP Tool Attribution

When using MCP tools to create entities on behalf of an agent:

```javascript
// CORRECT — agent identity attributed
mcp__epic-flowstate__collection-create({
  collection: "documents",
  orgId: "org_xxx",
  data: {
    title: "Implementation Plan",
    userId: "team_BySuGAjoaD" // From agent's teamMemberId
    // ...
  }
})

// WRONG — no attribution
mcp__epic-flowstate__collection-create({
  collection: "documents",
  orgId: "org_xxx",
  data: {
    title: "Implementation Plan"
    // userId missing — WHO created this?
  }
})
```

## Red Flags — STOP

- Dispatching a subagent without resolving its identity from D1 first
- **Reading `.claude/agents/<slug>.md` to extract a teamMemberId** — that file is generated and may lag behind D1. Always query teammembers.slug.
- **Reading `.flowstate/agents/<slug>.json` to extract a teamMemberId** — same problem. The JSON is the bootstrap definition, not the runtime source of truth.
- Creating entities without `userId` set to `record.id` (the teammember's `id`)
- Using a generic/anonymous agent when a defined agent exists for the task type
- Hardcoding `teamMemberId` values in CLI flags or code — pass slug to `--name`, let `resolveAgentRecord` do the lookup
- Resolved record has `isAgent=false` — slug matched a human teammember, do not dispatch as agent
- `resolveAgentRecord` returns multiple rows for the same slug — backfill missed deduplication; investigate before proceeding
- Multiple agents creating entities with the same `userId` — each agent has its OWN teamMemberId

## Selecting the Right Agent

When multiple agents could handle a task:

1. **Check capabilities** — does the agent's `capabilities` array include the needed step kind?
2. **Check org hierarchy** — is the agent at the right level? (ICs implement, managers review, directors approve)
3. **Check tools** — does the agent have the tools needed? (code writers need Edit/Write, reviewers need Read-only)
4. **Check model** — is the task complex enough for opus, or is sonnet sufficient?
