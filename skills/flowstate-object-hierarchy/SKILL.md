# FlowState Object Hierarchy

**Status:** Active
**Purpose:** Reference guide for the FlowState object model — hierarchy, dependencies, and create-if-missing rules
**Scope:** All processes that create or reference FlowState entities

---

## Absolute Rule: IDs Are Never Fabricated

**Every ID in FlowState is auto-generated by MCP tools.** Agents NEVER manually construct, guess, infer, or recall IDs from memory. This applies to ALL entity types without exception: `org_*`, `work_*`, `code_*`, `proj_*`, `mile_*`, `task_*`, `goal_*`, `docu_*`, `rec_*`, `prod_*`, `bpln_*`, and any other prefixed ID.

IDs come from exactly three sources:
1. **Config files** (`.flowstate/config.json`) for `orgId`, `workspaceId`, `codebaseId`, `projectId`
2. **MCP tool responses** (returned when creating or querying entities)
3. **The user** (when asked explicitly because the ID cannot be found)

If you cannot obtain an ID from sources 1 or 2, ask the user. Never proceed with a made-up value.

---

## Overview

FlowState entities follow a strict parent-child hierarchy. Every entity requires valid parent IDs. Before creating any entity, you must verify that all parent entities exist — and create them if they are missing.

```
┌─────────────────────────────────────────────────────────────┐
│  Org  (org_XXXXX)                                           │
│  └── Workspace  (work_XXXXX)                                │
│      ├── Codebase  (code_XXXXX)  [1 per git repo]           │
│      └── Project  (proj_XXXXX)   [1 per package/app]        │
│          └── Milestone  (mile_XXXXX)                        │
│              └── Task  (task_XXXXX)                         │
└─────────────────────────────────────────────────────────────┘
```

## Config File Mapping

### Root Repository Config (`.flowstate/config.json`)

```json
{
  "version": "1.0.0",
  "projectName": "<repo-name>",
  "projectType": "monorepo",
  "orgId": "org_9f3omFEY2H",
  "workspaceId": "work_ojk4TWK5D2",
  "codebaseId": "code_TDgAohsyX1",
  "vault": { "name": "flowstate-development", "type": "1password" },
  "repository": { "url": "<github-url>", "provider": "github" }
}
```

**Does NOT contain `projectId`** — that belongs at the package level.

### Per-Package Config (`packages/<pkg>/.flowstate/config.json`)

```json
{
  "orgId": "org_9f3omFEY2H",
  "workspaceId": "work_ojk4TWK5D2",
  "codebaseId": "code_TDgAohsyX1",
  "projectId": "proj_HpfQyaJBl5",
  "projectName": "@epicdm/flowstate-cli",
  "projectType": "cli"
}
```

**Contains `projectId`** — unique per package.

### Which IDs Go Where

| ID            | Root Config | Package Config  | What It Links          |
| ------------- | ----------- | --------------- | ---------------------- |
| `orgId`       | ✓           | ✓ (inherited)   | FlowState organization |
| `workspaceId` | ✓           | ✓ (inherited)   | Repository workspace   |
| `codebaseId`  | ✓           | ✓ (inherited)   | Git repository record  |
| `projectId`   | ✗           | ✓ (per package) | Package project record |

## Monorepo Package Rules

In a monorepo, **every npm package** that is tracked in FlowState must have:

1. A `Project` record in FlowState (`proj_XXXXX`)
2. A `.flowstate/config.json` with that `projectId`

All packages in the same monorepo share the same `orgId`, `workspaceId`, and `codebaseId`.

```
epic-flowstate/
├── .flowstate/config.json           ← orgId, workspaceId, codebaseId (NO projectId)
└── packages/
    ├── flowstate-cli/
    │   └── .flowstate/config.json   ← + projectId: "proj_HpfQyaJBl5"
    ├── flowstate-mcp/
    │   └── .flowstate/config.json   ← + projectId: "proj_ubTez4-mxO"
    └── flowstate-app-nextjs/
        └── .flowstate/config.json   ← + projectId: "proj_OC7RPZ8wZD"
```

## Pre-Flight Check: Task Creation

Before creating a task (the most common operation), verify the full chain:

```
1. READ .flowstate/config.json (package-level)
   → orgId, workspaceId, codebaseId, projectId

2. VERIFY org exists:
   collection-get orgs <orgId>
   → FAIL = invalid orgId, fix config

3. VERIFY workspace exists:
   collection-get workspaces <workspaceId>
   → MISSING = create workspace (workspace-registration-process.md)
             + update root config

4. VERIFY project exists:
   collection-get projects <projectId>
   → MISSING = create project (project-schema.md)
             + write projectId to package config
             + commit config

5. VERIFY or CREATE milestone:
   collection-query milestones { projectId, name: <group> }
   → MISSING = create milestone (milestone-schema.md)

6. CREATE task:
   collection-create tasks { ..., projectId, milestoneId, orgId, workspaceId }
```

## Quick Reference: Create If Missing

### If workspace is missing

```
1. collection-create workspaces { name, title, orgId, ... }
2. Write workspaceId to .flowstate/config.json
3. collection-create codebases { ..., orgId, workspaceId }
4. Write codebaseId to .flowstate/config.json
5. git commit .flowstate/config.json
```

### If codebase is missing

```
1. collection-create codebases { name, title, repository, environment, orgId, workspaceId }
2. Write codebaseId to root and all package .flowstate/config.json files
3. git commit .flowstate/**/config.json
```

### If project is missing

```
1. Check if projectId exists in package .flowstate/config.json
   → If yes: collection-create projects { id: <existing-id>, name, title, orgId, workspaceId }
   → If no:  collection-create projects { name, title, orgId, workspaceId }
             Write new projectId to package .flowstate/config.json
2. git commit packages/<pkg>/.flowstate/config.json
```

### If milestone is missing

```
1. collection-create milestones { name, title, projectId, goalId: "", orgId, workspaceId }
2. Record milestoneId for task creation
```
